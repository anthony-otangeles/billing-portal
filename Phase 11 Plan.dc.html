Phase 11 — The Portal Assistant

An AI assistant available on every billing screen that finds, explains, refines, and navigates — and never processes. It answers "where is…", "why did…", "show me all…", and "take me to…" questions from the same data every screen already shares. Anything that would change a claim is a hand-off to the governed surfaces (Agent Inbox, Payments), never an action the assistant takes itself.

Why this is a different thing from the agents you already have

The pipeline agents (generation, posting, follow-up) do the work and are governed by tiers, thresholds, and QA. This assistant is a navigator — a colleague who has read everything and can take you to the right place. Keeping it read-only by charter is what makes it safe to give it the whole platform as context: it can see everything the signed-in user can see, because the only thing it can do with that knowledge is show it to you, with citations.

11.1

The assistant panel — one shell, every screen

effort M
  • Entry: an "Ask" button in the top bar of every billing screen (beside search) + a keyboard shortcut. Opens a docked right-side panel — the page stays visible and usable, because the assistant's whole job is pointing at it.
  • Built like the other shared components (one web component, dropped into each header), conversation persists per session, works identically on all 11 screens.
  • Suggested prompts on open, seeded per screen ("What's holding up the Duran claim?", "Which payers are underpaying this month?") so the first use teaches the range.
  • Charter shown in the panel: "I find, explain, and navigate. I don't approve, edit, or submit — those stay in your queues."
11.2

Grounded answers — retrieval over the shared stores

effort M–L · the substance
  • Four question shapes, all against live store data (the same claims, contracts, and decisions every screen renders): Find ("claims for Rosa Whitfield", "everything sitting with secondary payers over $100") · Aggregate ("how much is Humana underpaying us?", "open AR by facility") · Explain ("why was this paid $125.70?", "what does CO-253 mean?" — drawing on the drift narratives, variance dispositions, and glossary) · Navigate ("take me to the MHS rate review").
  • Every answer cites its sources: claim chips, payer chips, and screen links you can click — the answer IS a set of deep links (Claim Detail, Register pre-filtered, Contracts with the code loaded, the exact Inbox decision).
  • Honest misses: when it can't answer from the data, it says so and offers the nearest thing it can show — never invents a number.
11.3

Refinement — a working set you narrow by talking

effort M
  • Follow-ups refine, not restart: "claims denied this month" → "just Humana" → "over $200" — the panel shows the current filter set as chips, with the count and dollars updating each turn.
  • Exit ramps into the real surfaces: "Open these in the Register" (lands with those filters applied), "Save this as a view" (10.3's saved views), "Any of these have open decisions?" → links to the Inbox group.
  • This is where it beats the search box: search finds a thing; the assistant carves a population.
11.4

Guardrails, worn on the outside

effort S · the trust story
  • Read-only by construction: the assistant has no approve/edit/submit verbs. Asked to act ("write off these claims"), it declines and links to the governed path — the decline is part of the demo.
  • Sees what you see, nothing more: scoped to the signed-in role and selected practice, same as every screen. Angela's assistant can't see Westbrook if Angela can't.
  • Auditable: assistant sessions are noted in the Audit Log the same way Ops portal entries are — a question trail, not a black box.

Decisions I need from you

  • Live AI or scripted intents? I can build it two ways. (a) Deterministic: a real retrieval engine over the stores with a curated intent parser — every demo question works perfectly, every time, offline. (b) Live Claude-powered: the panel actually calls Claude to interpret free-form questions against the store data — genuinely open-ended, more impressive, slightly less predictable in a demo. My recommendation: (a) as the foundation, with (b) as a toggle — the demo stays safe, the wow is available.
  • Name and voice? The pipeline agents are named by function ("Payment Reconciliation Agent"). Options: keep it functional ("Assistant"), or give it a light product name. I'd keep it functional — this platform's credibility is its restraint.
  • Where does it live? Docked right panel (my recommendation — keeps the page visible for navigation) vs. a centered command-palette overlay.
  • Scope of the first build: the four question shapes on the billing portal only, or also the Practice window (practices asking "what do you need from me?") — I'd hold the practice side for a later phase.
