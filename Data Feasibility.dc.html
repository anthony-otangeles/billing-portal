Data Feasibility: Billing Portal

Can the data behind every Billing Portal feature actually be obtained? Yes. Most of it arrives through one clearinghouse integration; the rest is either native to the platform, operationally maintained, or produced by the agent itself. Three items are genuinely hard and are flagged honestly below.

1 · The four source classes

A · Native — we generate it

Encounters, codes, charges, patients, facilities come from Otangeles itself. Everything the agent produces — decisions, touch logs, call summaries, learned rules — is our own data. Zero integration risk.

B · Clearinghouse EDI — one vendor

Standard X12 transactions via one API vendor (Claim.MD, Availity, Optum, Stedi, Waystar): 837 submission, 277CA accept/reject, 276/277 status, 835 payments + CARC/RARC, 270/271 eligibility and benefits.

C · Operationally maintained

No feed exists: payer phone numbers, IVR paths, hours, filing limits. A curated payer directory table someone owns and updates. Buildable on day one; a staffing commitment, not an integration.

D · Agent-worked

Data no API returns: payer-rep phone confirmations, portal-only pending detail, COB primacy rulings. The agent obtains it by call or portal — exactly what the Agent Inbox and Coverage screens already depict.

3 · The three honest flags

Prior authorization

X12 278 exists but payer adoption is poor. Realistically portal / phone / fax — the Coverage screen's auth check will be part-automated, part-agent-worked for the foreseeable future. Design already models this correctly.

COB primacy

271 sometimes flags other coverage, but confirming payer order (e.g. Medicare Benefits Coordination Center) is a phone/portal workflow, not an API. The Coverage board's "agent re-checking, ~48h" pattern is the accurate representation.

Legacy AR backfill

Claims billed before Otangeles (from another PM system) need an 837/835 backfill from the practice's prior clearinghouse or a PM export. Scope decision required: is pre-Otangeles AR in or out for launch?

5 · Integration decisions to make

  • Pick the clearinghouse vendor. Evaluate on: real-time 270/271 payer coverage for IN/MI Medicaid + MA plans, 835 enrollment turnaround, API quality, per-transaction pricing. This one contract powers ~70% of the portal.
  • Own the payer directory. Decide who maintains phones/IVR/filing limits — ops team, the billing companies themselves, or shared. Ship it as an editable table either way.
  • Insurance discovery add-on. Demographic-sweep eligibility ("Not our insured" cases) is sold as a product by several vendors — buy, don't build.
  • Portal RPA — defer. Payer-portal scraping vendors exist but add fragility; the agent's phone workflow covers the same gap. Revisit if call volume becomes the bottleneck.
  • Legacy AR scope (flag 3): decide before onboarding any practice with existing AR.
